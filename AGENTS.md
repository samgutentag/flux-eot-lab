# AGENTS.md

Guidance for an AI agent (Claude Code, Cursor, and the like) working with this repo.

## What this is

A pre-production lab that measures Deepgram Flux turn **detection** latency vs false-cutoff
across acoustic conditions and recommends an `eot_threshold` per condition. It tunes the one
span the knob controls: true end of speech to the first end-of-turn event.

## Read first

- **`METHODOLOGY.md`** is the domain model. Ground on it before reasoning about tuning: the
  knob semantics, the latency-vs-false-cutoff tradeoff, why p50/p95, the budget band, and a
  glossary. This is the file to read if you are asked to interpret results or suggest settings.
- **`README.md`** is how to run it.

## The rules that matter (do not break these)

- **Detection only.** The downstream LLM and text-to-speech pipeline is out of scope and is
  never in the measured path. Its variance is not tunable by `eot_threshold`.
- **No model in the measurement.** The harness is deterministic on purpose; reproducibility is
  the credibility spine. Do not introduce a model, sampling, or nondeterminism into capture or
  analysis.
- **Mock can never pass as real.** Every mock `TurnRecord` sets `mock=True`; mock analysis
  writes `recommendations_mock.md` and prints a banner; measured cells stay `PENDING_REAL_RUN`
  until a real keyed run fills them. Never present mock numbers as measurements.
- **Verified knob ranges live in `flux_eot_lab/config.py`** (`eot_threshold` 0.5-0.9,
  `eager_eot_threshold` 0.3-0.9 and <= `eot_threshold`, `eot_timeout_ms` 500-10000). Do not
  invent ranges; these are from Deepgram's Flux docs.

## Layout

- `bench.py` sweeps the knobs over the clips and writes raw per-turn JSONL.
- `recommend.py` is the `flux-bench` front door: it chains the sweep and the analysis.
- `flux_eot_lab/` is the package: `agent` (instrumented Flux agent), `analysis` (classify +
  recommend + render), `clients` (real Flux source), `mock` (deterministic fake source),
  `config`, `records`.
- `audio/` generates and annotates clips; `audio/clips/` holds the committed sidecars (ground
  truth); WAVs are gitignored and regenerated by `audio/generate_audio.py`.
- `results/` holds the committed evidence: `raw_real.jsonl` and `recommendations.md`.
- `skills/scaffold-test-clips/` is the one reusable skill: it authors new test conditions.

## Run it

- Mock (no key, no audio): `flux-bench --mock`
- Real run: `python audio/generate_audio.py` (regenerates the WAVs, needs `DEEPGRAM_API_KEY`),
  then `flux-bench --input audio/manifest.json --output results/`.

## Add a new test condition

Use `skills/scaffold-test-clips` to author clip definitions from a plain-language brief, then:

```bash
python audio/generate_audio.py --defs audio/generated/<condition>_defs.json
flux-bench --class <condition>
```

The model writes the dialogue; Aura synthesizes it, so `true_end_ms` is exact by construction.
The model never touches the ground truth or a measured number.
